BloodStrike Anti Cheat Vulnerability Research

AUTHOR: Matthew Dominguez • DATE: March 17, 2026

The game BloodStrike by NetEase Games was recommended to me by a friend. Something I am very interested in about every game I play is the game engine and the anti-cheat. When loading up bloodstrike I not only didn't see any popular game engine logos (like Unity or Unreal Engine), but I also didn't recognize anything anti-cheat related. After a short period of time in IDA Pro it's very easy to find the name "Messiah" appear in strings of the main binary of the game. Why are these strings not cleared? Not quite sure. Anyways, after reversing it's some of it's anti-cheat attempts, here is what I've found:

Internal Cheat Detection Related Observations:

1. Checking if d3d11.dll pointers point outside of d3d11.dll address range.

2. Checking if d3d11.dll WinVerifyTrust returns non-zero.

3. Checking if IsDebuggerPresent returns non-zero.

4. Hooking ReadFile, WSASend, WSARecv, and PyEval_EvalCodeEx, D3DCompile to monitor how many times each function has been called as well as how many bytes have been transferred in the case of ReadFile. Note, the D3DCompile hook function literally does nothing.

5. VirtualQuery is used in only one active function to scan the virtual memory of the game. It will completely skip over memory regions that are not MEM_COMMIT, that are MEM_PRIVATE, that are exactly PAGE_SIZE, or that are not executable. It creates a python list object and fills it with the regions it thinks are suspicious.

6. Calls K32GetProcessMemoryInfo and logs the WorkingSetSize. This can be used to compare to the amount of memory they are querying in the future to see if they are actually querying the entire virtual address space.

7. Module32FirstW / Module32NextW is used to create a list of module names, paths, base addresses, and sizes.

External Cheat Detection Related Observations:

1. Enumerating all processes and grabbing the module name of the first module, any window names, the process id, and an MD5 hash of the file on disk. Some things to note is that if OpenProcess fails, the failure is not tracked (aside from the entry being missing in the log). Another thing to notice is that it is only targeting the first module of the program.

2. Thread32First / Thread32Next is used to loop through all the threads on the system, but only storing the thread id. In other points NtQueryInformationThread is called to fetch things like the start address of the thread and the module who it belongs to.

Device Finger Printing:

1. Accessing dwOemId and dwNumberOfProcessors from GetNativeSystemInfo.

2. DeviceIoControl is called targeting "\\\\.\\PhysicalDrive0" through "\\\\.\\PhysicalDrive3".

Misc:

1. Fetches the file path of the main game module, they might be checking the files in the directory who knows.

This is not ALL of the features, only the ones I've analyzed.

After reversing all of these functions. I've noticed there are critical vulnerabilities in the anti-cheat. Here are some:

They completely ignore external processes who cannot have a handle with PROCESS_ALL_ACCESS opened. They don't even log the failures like I said above. A program could easily set protection settings that disallow for this type of handle from being opened and completely avoid being investigated by the anti-cheat.

They also are only interested in the first module of external applications, they completely ignore the dlls loaded. If an external cheat is compiled as a dll and injected into a legitimate process, it will completely be unseen.

Another issue is that they are hashing the entire file on disk as opposed to specific sections like the .text or better yet maybe things like file metadata. In the current state, the hash will be completely different by a single bit being flipped anywhere in the binary on disk.

During their scan for suspicious memory ranges within their address space. They completely skip over ranges that don't meet a ridiculously specific criteria, including MEM_PRIVATE ranges. Meaning a threat actor can just set their allocation to be MEM_PRIVATE and go completely under the radar.

Though there are plenty more things that could be mocked in this anti-cheat, I will leave it with the last major vulnerability which is one of the most important aspects of all anti-cheats. Which is code obfuscation. They made no attempt in obfuscating any of this code which just allows any random person who knows how to use a disassembler to take apart this anti-cheat.

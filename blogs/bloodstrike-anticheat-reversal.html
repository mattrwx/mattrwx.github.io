BloodStrike Anti Cheat Vulnerability Research

AUTHOR: Matthew Dominguez • DATE: March 17, 2026

The game BloodStrike by NetEase Games was recommended to me by a friend. Something I am very interested in about every game I play is the game engine and the anti-cheat. When loading up bloodstrike I not only didn't see any popular game engine logos (like Unity or Unreal Engine), but I also didn't recognize anything anti-cheat related. After a short period of time in IDA Pro it's very easy to find the name "Messiah" appear in strings of the main binary of the game. Why are these strings not cleared? Not quite sure. But it give's a good look at the structure of the game engine and many interesting classes that I may or may not reverse in the future. I know the game has an anti-cheat (because I was banned for using a well known 3rd party dynamic analysis tool). But I don't know where this anti-cheat lives. Most modern anti-cheats are not a part of the main binary of the game, but are loaded in during runtime as a dll. However, thanks to the brilliant minds at NetEase, the anti-cheat is right there embedded in the main binary. And, in case your expectations aren't already low, let me lower them further, it's completely unprotected; Practically open source.

BOOL Hook_ReadFile()
{
    HMODULE ModuleHandleW = GetModuleHandleW(L"Kernel32.dll");
    FARPROC ReadFile_addr = GetProcAddress(ModuleHandleW, "ReadFile");

    std::printf("Hook ReadFile, Addr = 0x%p\n", ReadFile_addr);

    DWORD flOldProtect;
    VirtualProtect(ReadFile_addr, 0xC, PAGE_EXECUTE_READWRITE, &flOldProtect);

    read_file_hook.targetAddress = ReadFile_addr;
    read_file_hook.originalBytes = *(std::uint64_t*)ReadFile_addr;
    read_file_hook.originalNext = *((std::uint64_t*)ReadFile_addr + 2);
    read_file_hook.stubMovRax = 0xB848;
    read_file_hook.hookAddress = ReadFile_hk;
    read_file_hook.stubPushRet = 0xC350;
    
    /*
    mov rax, ReadFile_hk
    push rax
    ret
    */

    return WriteProcessMemory(GetCurrentProcess(), ReadFile_addr, &read_file_hook.stubMovRax, 0xC, 0LL);
}

So, in the code above you will see how the anti-cheat has decided to place some hooks. It will overwrite the start of the function it is targeting (in this case ReadFile). But majority of the time code is going to be in a read only section, so the function's page's protection to PAGE_EXECUTE_READWRITE. And byte patching the start of the function to the asm in the comment below. If you read the ReadFile_hk function, we can see how they execute the original function. It repairs the function, calls it, saves the return and then patches it again. This is a questionable approach considering it creates a race condition that would make the program unstable if ReadFile is called around the same time in two seperate threads. And the mutex doesn't eliminate this race condition, it only gives better odds. The purpose of this whole hook is to just count how often the function is called and how many bytes are read. This type of hook is used on these functions:

  • ReadFile (kernel32.dll)
  • WSARecv (ws2_32.dll)
  • WSASend (ws2_32.dll)
  • PyEval_EvalCodeEx (PythonCore_x64h.dll)

The next interesting thing I investigated was a form of hardware tracing. Taking a look at the disassembly generated by IDA Pro, you can see a series of string literals being fed into some array. Then a CreateFileW call which is how a connection can be established from usermode to a driver through IOCTL. We know this is the disk driver becuase of the strings we just saw. Next, we see a DeviceIoControl call to this driver. This is cut and clear, this is fetching some sort of information off of disk driver.

Someone trying to defeat this tracing functionality might thing of well known methods for disk spoofing like redirecting the IOCTL calls to this driver to a custom driver that intercepts these IOCTL requests before deciding to forward them to the driver or handle them itself. However, I think this may be overkill. There is hooking being done on the DeviceIoControl function. And I have yet to find any detection mechanisms responsible for hashing and validating the modules. So by simply hooking the DeviceIoControl function you can avoid the syscall as a whole and just return fabricated information. This goes for ALL of the DeviceIoControl checks.

Also, as a side note, this is atrocious programming. Why are we hardcoding the paths for disk drive 0-3, have we not yet learned what a for loop is?

Next, let's take a look at one of their functions that is supposed to scan external processes. This function is a masterpiece. I've rebuilt this in C++ to make it clear, but this function basically uses, EnumProcesses. Then looping through those processes. In this loop they are ensuring they can open a handle with PROCESS_ALL_ACCESS. Checking if EnumProcessModules fails when they try to fetch the first module. And lastly, checking if GetModuleBaseNameW on that first module fails. It saves this name for later. However, if your process fails these checks, it will simply skip over you. Once it finds a valid external process to investigate, it will break out of the loop (meaning this is the only process it will investigate) and do some stuff like hashing, checking window names, etc.

Whoever wrote this function needs to be fired immediately. It would be a struggle to even get your process to get investigated. There are so many ways to avoid getting looked at, all you have to do is just not let your program pass the first set of tests. The easiest way would be to just disallow for a PROCESS_ALL_ACCESS handle to be opened to your program with DACL manipulation.
